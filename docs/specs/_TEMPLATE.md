# Spec: <feature-name>

> **生成 Spec 唯一入口为 `docs/specs/`。Spec 是功能的顶层设计，只描述 WHAT 和 WHY，不描述 HOW。**

## 1. 要解决的问题

用 2-3 句话描述用户当前无法做什么，为什么这是一个问题。

## 2. 功能范围

- 包含什么
- 不包含什么（显式划边界）

## 3. 技术选型（如有分歧才写）

如果有多种实现路径，在此列出并说明选择了哪条、为什么。没有分歧则删除此节。

## 4. 验收标准

每条 AC 必须**可验证**——agent 能直接翻译为测试函数。

- [ ] AC-1: <描述> → 对应测试 `test_xxx`
- [ ] AC-2: <描述> → 对应测试 `test_xxx`
- [ ] AC-3: <边缘情况> → 对应测试 `test_xxx`

## 5. 依赖

- 依赖哪些已有 feature/module
- 哪些后续 feature 依赖本 spec

## 6. 风险与假设

- 技术上不确定的点
- 对用户行为的假设
