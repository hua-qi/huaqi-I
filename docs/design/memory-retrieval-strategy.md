# 记忆检索策略

## 概述

huaqi 的核心体验依赖「记住用户」。随着对话历史、日记、用户画像持续积累，如何在每轮对话时高效找到相关历史内容，是影响回答质量和系统可持续性的关键设计决策。

本文档记录各方案的分析、对比与最终选型，作为实现参考。架构决策详见 [ADR-003](../../spec/decisions/ADR-003-memory-retrieval-strategy.md)。

---

## 数据量增长预估

基准假设：用户每天 5 轮对话 × 2 次会话，1 篇日记。

| 时间节点 | 对话文件数 | 日记数 | 向量条目数 | 文本总量 |
|------|------|------|------|------|
| 1 个月 | ~60 | 30 | ~300 | ~200KB |
| 6 个月 | ~360 | 180 | ~1,800 | ~1.2MB |
| 1 年 | ~720 | 365 | ~3,600 | ~2.5MB |
| 3 年 | ~2,160 | 1,095 | ~10,800 | ~7.5MB |

单条 Chroma 向量记录约 1.7-2KB（内容 + 384 维向量 + 元数据）。

---

## 方案详细分析

### 方案一：大模型直接检索

**原理**：将历史数据全量塞入 prompt，让 LLM 自行定位相关内容。

**随数据量增长的表现**：

```
1 个月（~50K tokens）
  GPT-4o 支持 128K，技术上可行
  成本：~$0.25/次对话，不可接受

6 个月（~300K tokens）
  超过多数模型 context window，必须截断
  截断策略本身就是一种检索，绕回原点

3 年
  完全失效，约 375 万 tokens，任何模型都放不下
  即使放得下，LLM「lost in the middle」问题会导致中间内容被严重忽略
```

**结论**：不可持续，短期成本已不可接受。

---

### 方案二：LLM 关键词提取 → BM25 全文检索

**原理**：用小 prompt 让 LLM 从用户输入提取关键词，再用 BM25 在全量文件中字面匹配。

**核心缺陷**：

```
用户日记写：「今天情绪很低落」
LLM 提取关键词：「焦虑」「心情差」
BM25 结果：0 条命中（字面不同）

用户问：「最近怎么样？」（泛化问题）
LLM 提取关键词：「最近」「状态」
BM25 结果：300+ 条命中（噪音失控）
```

**随数据量增长的延迟**：

| 时间节点 | BM25 全量扫描延迟 |
|------|------|
| 1 个月 | <50ms |
| 1 年 | ~500ms |
| 3 年 | 2-5 秒（破坏流式体验）|

**变体价值**：当搜索对象从原始文件换成「结构化周/月摘要」时，关键词和摘要用词天然对齐，局限性大幅缩小。适合在 L4 层（3 个月以上历史）作为摘要索引的检索手段。

---

### 方案三：Embedding + 向量检索（BM25 混合）★ 当前选型

**原理**：离线将历史数据向量化存入 Chroma，查询时语义召回 top-K 最相关片段。

**项目现状**：基础设施已完备，仅需激活。

```python
# chat_nodes.py 当前（未激活向量）
search = get_hybrid_search(use_vector=False, use_bm25=True)

# 激活后
search = get_hybrid_search(use_vector=True, use_bm25=True)
```

融合分数公式：

```
final_score = 0.7 × vector_score + 0.3 × bm25_score + recency_boost
recency_boost = exp(-0.1 × days_ago)
```

**随数据量增长的表现**：

| 时间节点 | 检索延迟 | 向量条目数 | 磁盘占用 |
|------|------|------|------|
| 1 个月 | <10ms（内存索引）| ~300 | 0.5MB |
| 6 个月 | <50ms | ~1,800 | 3MB |
| 1 年 | <100ms | ~3,600 | 6MB |
| 3 年 | <200ms（HNSW）| ~10,800 | 18MB |

**3 年后的潜在挑战**：
- 向量粒度问题：按文件索引粒度过粗，细节查询精度下降
- 解决方案：按对话回合分块重新索引（届时再处理，YAGNI）

**Embedding 模型**：`BAAI/bge-small-zh`（512 维，本地运行，首次启动自动下载 ~100MB）

---

### 方案四：层级摘要索引

**原理**：每天/周/月自动生成摘要，检索时只查摘要树。

```
原始对话/日记 → 日摘要（~100 tokens）→ 周摘要（~200 tokens）→ 月摘要（~300 tokens）
```

**适合**：宏观查询（「这个月主要在做什么」「最近状态怎样」）。
**局限**：信息有损耗，无法回答细节问题。
**引入时机**：作为 L4 层（3 个月以上历史）的宏观索引，与方案三组合使用。

---

### 方案五：知识图谱（GraphRAG）

**原理**：从文本中提取实体和关系构建图，通过图遍历回答关系型查询。

```
[用户] —— 在做 ——→ [Kaleido项目] —— 遇到 ——→ [灰度发布卡点]
[用户] —— 学习 ——→ [电吉他] —— 上次练到 ——→ [第3课·拨弦]
```

**适合**：关系型查询（「我和 XX 项目的关系」「哪个目标进展最慢」）。
**局限**：情绪/心理类信息难以图化；实现复杂度最高。
**引入时机**：数据量积累到 1 年以上再评估。

---

### 方案六：结构化提取 + SQL 查询

**原理**：每次对话结束后 LLM 提取结构化字段写入 SQLite。

```sql
-- 示例查询：情绪低落的时间点
SELECT date, key_events FROM daily_records
WHERE mood = '焦虑' ORDER BY date DESC LIMIT 5
```

**适合**：情绪统计、习惯打卡、目标进度聚合查询。
**局限**：字段设计决定能查什么，扩展性受限。
**引入时机**：作为情绪/统计类查询的专用层，与向量检索并列。

---

### 方案七：HyDE（假设文档嵌入）

**原理**：查询时先让 LLM 生成「假设的历史记录」文本，用这段文本的向量代替原始问句向量做检索，解决问句短、文档长的向量不对称问题。

**定位**：方案三的增强手段。向量召回率不足时按需启用。代价是多一次 LLM 调用（+200ms，+~150 tokens）。

---

### 方案八：粗召回 + Re-ranking 精排

**原理**：BM25 或向量粗召回 Top-20，Cross-Encoder 精排至 Top-3。

**定位**：方案三的增强层。Cross-Encoder 本地运行，无 API 成本，+50~200ms 延迟。**引入时机**：召回精度出现明显问题时再评估。

## 各方案综合对比

| 方案 | 1年精度 | 3年延迟 | LLM成本 | 实现复杂度 | 适合查询类型 |
|------|------|------|------|------|------|
| 方案一：LLM直接 | 失效 | 不可用 | 极高 | 低 | 全类型（短期） |
| 方案二：关键词+BM25 | 中 | 2-5s | 低 | 中 | 精确字面查询 |
| **方案三：Embedding** ★ | **高** | **<200ms** | **0** | **中** | **语义查询** |
| 方案四：摘要索引 | 中 | <50ms | 低（离线） | 中 | 宏观时间线 |
| 方案五：知识图谱 | 高 | <10ms | 低（离线） | 高 | 关系型查询 |
| 方案六：结构化SQL | 高 | <5ms | 低（离线） | 低 | 统计/趋势 |
| 方案七：HyDE | 高 | +200ms | 低 | 低（增强层） | 召回率补强 |
| 方案八：Re-ranking | 高 | +100ms | 0 | 低（增强层） | 精度补强 |

## 最终架构：分层混合检索

不同查询类型路由到不同检索层，各层独立可升级：

```
L1  当前会话（in-memory）
    └─ LangGraph AgentState.messages
    └─ 覆盖：「刚才说的那个...」

L2  近 7 天（直接读文件）
    └─ 读 conversations/ + diary/ 近 7 天 Markdown
    └─ ~5K tokens，直接注入 system_prompt
    └─ 覆盖：「今天/这周...」

L3  7 天 ~ 3 个月（向量检索）★ 当前优先实现
    └─ Embedding（bge-small-zh）+ BM25 混合
    └─ Chroma 向量库，top-3 召回
    └─ 覆盖：「上次提到的...」「之前聊过的...」

L4  3 个月以上（摘要索引，远期）
    └─ LLM 关键词 → 搜索周/月摘要
    └─ 摘要数量可控（3年 ≈ 156 周摘要）
    └─ 覆盖：「去年大概在做什么」
```

**当下实现范围**：L1（已有）+ L2（已有）+ L3（激活向量检索 + 补全 save_conversation 写入 Chroma）

**近期补充**：方案六（结构化 SQL）支持情绪/习惯统计

**远期评估**：方案四（L4 摘要索引）、方案五（知识图谱）、方案七/八（精度增强）
