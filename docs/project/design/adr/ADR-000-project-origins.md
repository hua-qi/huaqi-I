## 背景

本项目源于用户（连子蒙）想要打造一个专属的个人 AI 助手的需求。通过多轮对话，逐步明确了需求、探讨了方法论，并最终形成了系统的设计方案。

## 原始需求

用户希望 AI 助手具备以下能力：

1. **记忆**: 了解用户的身份、偏好、正在做的事情
2. **执行**: 信息收集、Coding、发布小红书、辅导电吉他、辅导英语等
3. **成长**: 随着了解深入和技能学习，能做的事情越来越多
4. **陪伴**: 答疑解惑、辅助决策、提升认知

## 技术方案决策

### 存储方案选择

经过对比，选择 **混合存储方案 (文本 + 向量)**：

```
Layer 1: 文本文件 (Markdown/YAML)    ← 人类可读、版本控制
Layer 2: 向量索引 (Chroma)           ← AI 语义检索
Layer 3: 云端备份 (Git)              ← 跨设备同步
```

**选择理由**:
- 文本文件保证数据的可读性和可维护性
- 向量索引支持语义检索
- Git 原生支持跨设备迁移
- 符合"可换公司/换电脑"的约束

### 架构设计

采用 **7 层架构** (基于 PAI 理念)：

```
7. 接口层    - CLI 为主 + 语音 + Web
6. 编排层    - Hook 系统 + 上下文管线
1. 智能层    - LLM + 持续学习
2. 上下文层  - 三层记忆系统
3. 个性层    - 可配置 AI 性格
4. 工具层    - MCP 技能生态
5. 安全层    - 多层防御 + Constitution
```

### 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 语言 | Python 3.10+ | AI 生态丰富 |
| CLI | Typer + Rich | 现代化、交互友好 |
| 向量库 | Chroma | 本地优先、零配置 |
| 配置 | Pydantic Settings | 类型安全 |
| 同步 | Git | 版本控制、跨设备 |

## 关键决策点

### 1. 为什么采用混合存储？

- 纯向量存储: 语义检索强，但不可读、难维护
- 纯文件存储: 可读、版本控制，但检索弱
- 混合方案: 兼顾两者，虽然架构复杂，但最符合需求

### 2. 为什么以 CLI 为主？

- 开发者友好
- 可集成到现有工作流
- 便于版本控制和自动化
- 未来可叠加 GUI/Web 层

### 3. 为什么先实现记忆系统？

- 是整个系统的"地基"
- 其他能力都依赖记忆
- 可以独立测试和迭代

### 4. 为什么使用 Git 同步？

- 用户已经在使用 GitHub
- 原生版本控制
- 支持离线使用
- 数据完全可控

## 设计理念总结

1. **Human 3.0**: 通过 AI 增强人类能力
2. **同伴关系**: 不是工具，而是共同成长的伙伴
3. **可迁移**: 换电脑/换公司不受影响
4. **可进化**: 系统可以学习和成长
5. **隐私优先**: 本地优先，用户完全控制
