# Feature: TELOS 蒸馏质量优化

> 实施日期：2026-05-18

## 解决的问题

763 条历史信号处理后发现的三个结构性问题：

1. **核心层空洞** — beliefs/models 仍为「（待补充）」，触发门槛过高
2. **维度坍缩** — 所有维度都在讨论同一件事（用户与 AI 的关系）
3. **版本膨胀/语言虚浮** — goals v97、learned v151，大量无意义重写，语言越来越抽象

## 实现方案

纯 prompt 优化，不涉及引擎代码改动。

### 改动点

| 文件 | 改动 |
|------|------|
| `step1.md` (data dir) | 核心层触发扩展为「隐含信念/行为选择/因果逻辑」；维度上限 2 个/信号；medium 积累可触发核心层初始化 |
| `step345.md` (data dir) | 关联点验证（必须指定具体信号）；new_content ≤300 字 + 必须引用信号；禁止元认知/跃迁/校准/共建/范式/涌现 |
| `_defaults.py` | 同步 step1 和 step345 内置回退 |

### 未改动

- `engine.py` — 已有 95% 相似度跳过 + history cap，本次不需要额外代码变更
- `manager.py` — 无需改动

## 关键设计结论

1. **Prompt 层软约束优于代码层硬约束**：代码的去重（95% 相似度跳过）已经做了，剩余问题是 LLM 输出质量，应从 prompt 端治本
2. **关联点验证是防坍缩的核心机制**：要求 LLM 在更新前指定「哪条信号 → 当前维度 → 为什么关联」，无法回答则跳过——这比单纯的「不要牵强」更有效
3. **禁词列表是治标不治本的权宜之计**：LLM 可能换其他抽象词继续。如果问题持续，应升级为 positive constraint（要求用具体场景描述而非抽象术语）
4. **核心层积累初始化是实验性规则**：允许 medium 信号累积触发 beliefs/models 初始化，降低了门槛但可能产生噪声——需要在后续迭代中观察
