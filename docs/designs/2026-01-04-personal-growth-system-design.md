# 个人成长系统 — 完整设计文档

---

## 1. 系统定位

### 核心定位

一个随用户成长的 AI 自我镜像。系统的价值随时间累积——用得越久，它越了解你，它给你的建议越有价值，它帮你干活越省力。

### 与 Miessler PAI 的关系

借鉴 PAI 的 TELOS 数据结构，但核心差异是把 TELOS 从「用户手动维护的静态文件」变成「Agent 动态维护的演化知识图谱」，并将成长本身（而不是效能）作为系统的第一目标。

| | Miessler PAI | 本系统 |
|---|---|---|
| AI 的角色 | 你的工具团队 | 你的成长伙伴 |
| 目标 | 效能最大化 | 自我实现 |
| 成长定义 | 能做更多事 | 成为更好的人 |
| TELOS | 用户手动维护的静态文件 | Agent 动态维护的演化知识图谱 |

## 2. 整体架构

### 三层 + Agent 驱动

```
┌──────────────────────────────────────────────────┐
│                    用户                           │
│            输入 / 对话 / 指令                     │
└─────────────────────┬────────────────────────────┘
                       │
┌─────────────────────▼────────────────────────────┐
│                    Agent                          │
│  Command（意图解析）                              │
│  Skill（能力编排）                                │
│  Tool（最小读写单元）                             │
│  MCP（外部数据接入）                              │
└───────┬──────────────┬──────────────┬────────────┘
        │              │              │
        ▼              ▼              ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│   数据层      │ │   成长层      │ │   能力层      │
│              │ │              │ │              │
│ RAW_SIGNAL   │ │ TELOS        │ │ 写作助手      │
│ SQLite       │ │ 成长事件      │ │ 定时报告      │
│ ChromaDB     │ │ 主动提问      │ │ 学习追踪      │
│ raw_files    │ │              │ │ 决策辅助      │
└──────────────┘ └──────────────┘ └──────────────┘
```

### 核心架构原则

1. **数据只增不减**：RAW_SIGNAL 永久保留，是系统的护城河
2. **用户主权**：TELOS 的创建/修改/删除由用户决定，Agent 只建议
3. **抽象接口优先**：所有 I/O 通过接口，本地/云端随时可切换
4. **最终一致性**：不依赖持续运行，保证最终处理
5. **全透明**：数据流向对用户完全可见
6. **一切皆数据**：用户的表达、行为、沉默都是 RAW_SIGNAL
7. **闭环飞轮**：每次输出写回数据层，每次交互加深系统对用户的理解

## 3. 数据层

### 职责

收、存、不丢。数据层不理解数据的含义，只负责可靠地存储。

### 存储结构

```
本地文件系统
├── TELOS/                    ← 成长层的 Markdown 文件（Git 管理）
├── raw_files/                ← 原始音视频/图片（永久归档）
│   └── 2026/01/04/
│       ├── voice_memo.m4a
│       └── meeting.mp4
└── .git/                     ← Git 版本管理，定期同步云端备份

数据库
├── SQLite（或云端 PostgreSQL）
│   └── raw_signals 表        ← 结构化时序数据
└── ChromaDB（或云端 Pinecone）
    └── 向量索引               ← 语义相似度检索
```

### RAW_SIGNAL 数据结构

```json
{
  "id": "uuid",
  "user_id": "用户 ID",
  "source_type": "journal | ai_chat | wechat | reading | calendar | audio | absence | ...",
  "timestamp": "ISO 8601",
  "content": "转换后的文本内容",
  "metadata": {
    "participants": [],
    "book_title": "",
    "raw_file_ref": "2026/01/04/voice_memo.m4a"
  },
  "processed": false,
  "distilled": false
}
```

### 数据源（可无限扩展）

所有数据类型经过转换器统一转成文本后进入 RAW_SIGNAL：

| 数据类型 | 转换方式 | source_type |
|---|---|---|
| 日记/笔记 | 直接摄取 | journal |
| AI 对话 | 直接摄取 | ai_chat |
| 聊天记录 | 解析导出文件 | wechat / telegram |
| 阅读笔记 | API / 导出文件 | reading |
| 音频 | Whisper 语音转文字 | audio |
| 视频 | 提取音轨 → Whisper | video |
| 图片 | OCR + 视觉描述 | image |
| 日历事件 | API 摄取 | calendar |
| 沉默期 | 系统自动生成 | absence |

**核心原则**：用户的一切表达和行为都是数据。不存在「没有输入」的状态，用户沉默本身也是信号。

### 三层记忆结构（蒸馏机制）

```
热记忆   最近 30 天的 RAW_SIGNAL    每次对话都可能用到
温记忆   蒸馏后的精华记忆            定期检索，长期保留
冷记忆   归档的原始 RAW_SIGNAL       按需检索，distilled=true
```

定期（每月）蒸馏：1000 条零散信号 → 10 条精华记忆，原始数据标记为已蒸馏但永不删除。

### 抽象接口

```
StorageAdapter
  ├── 本地实现（SQLite + 文件系统）
  └── 云端实现（PostgreSQL + S3）

VectorAdapter
  ├── 本地实现（ChromaDB）
  └── 云端实现（Pinecone）

SchedulerAdapter
  ├── 本地实现（cron）
  └── 云端实现（任务队列）

InterfaceAdapter
  ├── CLI 实现（初期）
  ├── Web 实现（后期）
  └── Mobile 实现（更后期）
```

### 备份策略

- TELOS（Markdown）、蒸馏精华记忆纳入 Git 版本管理
- 定期 push 到远端（GitHub / 私有仓库）
- Git 历史本身即成长时间线的另一种呈现

### 搜索

两个入口，共用同一套搜索接口：

```
对话触发   用户说「帮我找所有提到焦虑的记录」→ Agent 解析意图 → 调用搜索接口
API 搜索   直接调用搜索接口 → 方便外部工具集成、自动化脚本
```

## 4. 成长层

### 职责

理解、提炼、更新。成长层是 Agent 对用户认知的结构化表达，不存原始数据，只维护对用户的理解。

> 成长层的本质是一个持续运行的「理解引擎」——把原始的生活碎片，提炼成对你这个人越来越深刻的认知。

### 动态 TELOS

TELOS 是系统的心脏，维护一份关于用户的动态知识图谱。

**三个优先级，8+ 个维度：**

```
核心层（变化最慢，需要大量信号才触发更新）
  BELIEFS    — 信念和价值观
  MODELS     — 世界观和心智模型
  NARRATIVES — 关于自己的故事和自我认知

中间层（定期变化）
  GOALS      — 目标（短/中/长期）
  CHALLENGES — 正在面对的困难和卡点
  STRATEGIES — 应对问题的方式和习惯

表面层（频繁变化，Agent 自动更新）
  LEARNED    — 学到了什么（阅读/课程/经历提炼）
  PEOPLE     — 人际关系图谱
  SHADOWS    — 盲点和弱点（Agent 观察到但用户可能没意识到的）

特殊维度
  META       — 关于如何理解用户的偏好
               （提炼逻辑校正 + 活跃维度列表 + 维度演化历史）
```

**动态维度机制：**

- 8 个标准维度是起点，不是终点
- Agent 观察到标准维度之外的重复信号（≥N 条，跨越≥M 天）时，向用户提案新增维度
- 用户拥有完全主权：只有用户能创建/归档/删除维度，Agent 只能建议
- 归档的维度保留历史，不删除

**每条 TELOS 数据结构（Markdown + frontmatter）：**

```markdown
---
dimension: BELIEFS
content: 当前内容
updated_at: 2026-01-04
update_reason: 从日记中提炼：连续三次提到对「不确定性」的恐惧
confidence: 0.7
layer: core
---

历史版本记录...
```

### 信号提炼管道

```
RAW_SIGNAL（processed=false）
    ↓ Agent 定时检测
Step 1：单条信号分析
    这条内容涉及哪些 TELOS 维度？
    情感极性？强信号还是弱信号？
    ↓
Step 2：跨时间聚合
    过去 N 天，同一维度累积了多少信号？
    趋势是强化已有认知，还是在挑战它？
    ↓
Step 3：更新决策
    信号强度 × 累积次数 × 情感强度 > 阈值？
    ↓
Step 4：更新哪一层
    表面层 → 直接自动更新
    中间层 → 更新 + 轻量通知
    核心层 → 更新 + 生成变化报告
    ↓
Step 5：触发成长事件？
    TELOS 发生跳跃性变化 → 生成成长事件
    ↓
processed = true
```

**信号强弱判断：**

```
强信号（高可能触发 TELOS 更新）
  - 用户明确表达了立场
  - 情绪高峰/低谷时写的内容
  - 同一主题反复出现（≥3次）
  - 用户主动标注「这对我很重要」

弱信号（积累后才有意义）
  - 日常流水记录
  - 转述别人的观点
  - 单次提及某个想法
```

### 提炼逻辑的三个阶段

```
阶段 1（Day 1-30）   系统内置通用 prompt，适用所有用户
阶段 2（Day 30+）    Agent 提炼结论展示给用户，用户反馈写入 META
阶段 3（长期）       META 积累足够反馈，提炼 prompt 动态生成，完全个性化
```

### 成长事件

TELOS 某维度发生跳跃性更新时触发：

```json
{
  "id": "uuid",
  "user_id": "用户 ID",
  "timestamp": "2026-01-04",
  "dimension": "BELIEFS",
  "description": "你从「相信努力一定有回报」变成了「相信选择比努力更重要」",
  "trigger_signals": ["raw_signal_id_1", "raw_signal_id_2"],
  "before": "旧版本内容",
  "after": "新版本内容",
  "user_confirmed": null
}
```

### 主动提问

基于当前 TELOS 状态，Agent 定期生成深度问题——问出「用户自己不会问自己的问题」。

## 5. 能力层

### 职责

以 TELOS 为上下文的 AI 功能集合。核心差异化：同样的功能，因为了解你，所以更好用。能力层是插件，数据层+成长层才是护城河。

### 统一调用模式

```
输入（用户指令 or 定时触发）
    ↓
TELOS 注入 + ChromaDB 检索
    ↓
调用云端 AI
    ↓
输出结果
    ↓
结果写回 RAW_SIGNAL（流回数据层，形成闭环）
```

### 五个核心模块

**日常对话**
- 每次对话前自动注入 TELOS 上下文和相关历史
- 对话本身写入 RAW_SIGNAL

**定时报告**

| 报告 | 触发 | 内容 |
|---|---|---|
| 晨报 | 每天早上 | 基于 GOALS/CHALLENGES，今天值得关注什么 |
| 日报 | 每晚 | 当天 RAW_SIGNAL 提炼 + 一个反思问题 |
| 周报 | 每周日 | TELOS 变化 + 成长事件回顾 |
| 月报 | 月末 | 认知迭代总结 + 下月建议 |
| 年报 | 年末 | AI 叙事报告：这一年你发生了哪些内在变化 |

报告不是数据汇总，是 Agent 写给用户的信。

**写作助手**

| 任务 | 自动注入的 TELOS 维度 |
|---|---|
| 写晋升文档 | GOALS + LEARNED + NARRATIVES + 近期成长事件 |
| 写 PPT | STRATEGIES（表达风格）+ PEOPLE（受众）|
| 做决策分析 | BELIEFS + GOALS + CHALLENGES + 历史决策记录 |
| 制定学习计划 | GOALS + LEARNED（避免重复）+ STRATEGIES |
| 写邮件 | PEOPLE（对方是谁）+ NARRATIVES（表达风格）|

**学习追踪**（独立数据结构）

```
课程结构    学习路径（章节/主题/顺序）
进度状态    每个节点：未开始/进行中/完成
知识检验    小测验，判断是否真正掌握
间隔复习    遗忘曲线提醒
```

学习产生的内容照常流入 RAW_SIGNAL → 成长层，LEARNED 和 CHALLENGES 自动更新。

**决策辅助**

不给答案，给出「基于你是谁」的分析框架。查 BELIEFS/GOALS/PEOPLE/历史决策记录后生成个性化分析。

## 6. Agent

### 职责

数据层和成长层都是静止的数据，Agent 是驱动它们流转的引擎。

```
数据层  =  仓库（存什么）
成长层  =  大脑（理解什么）
Agent   =  那个一直在工作的人（做什么）
```

### 执行机制

```
Command   用户触发的指令（/journal /report /learn /decide）
Skill     能力层的每个模块（一组 Tool 的编排 + 对应 prompt）
Tool      操作数据的最小单元（read_raw_signal / write_telos / search_chromadb）
MCP       对外连接的桥梁（接入微信/日历/微信读书/Kindle）
```

### System Prompt 构成

每次对话动态组装四个部分：

```
Part 1：身份准则（静态）
  你是用户的个人成长伙伴
  你的回答始终基于你对他的认知
  你不是通用 AI，你是专属于他的 AI

Part 2：当前 TELOS 快照（动态，从成长层读取）
  BELIEFS / GOALS / CHALLENGES / ...

Part 3：相关历史记忆（动态，ChromaDB 按需检索）
  和当前对话语义相关的历史 RAW_SIGNAL 片段

Part 4：当前对话模式（动态，按触发场景切换）
  日常对话 / 写作助手 / 决策辅助 / 晨报...
```

### 三种工作模式

**后台观察者（持续被动）**
- 检测 processed=false 的 RAW_SIGNAL，运行信号提炼管道
- 表面层自动更新，核心层生成报告通知用户

**主动提问者（定期）**
- 读取当前 TELOS 状态，生成深度问题
- 发送定时报告（晨报/日报/周报/月报/年报）

**按需顾问（用户触发）**
- 构建 System Prompt → 调用云端 AI → 输出结果 → 写回 RAW_SIGNAL

### 运行模式

- **后台持续运行**：本地守护进程，开机自启
- **最终一致性**：设备关机时 RAW_SIGNAL 积压，恢复后批量处理
- **云端部署**：通过抽象接口切换，天然解决持续运行和多设备问题

## 8. 用户相关设计

### 多用户

每个用户有独立的数据隔离空间，Agent 运行时严格限定在当前用户的数据边界内。

### 冷启动

```
Step 1  引导问卷（10 个核心问题，20 分钟内建立初始 TELOS）
Step 2  历史数据冷导入（过去的日记、读书笔记、聊天记录）
Step 3  系统主动说明「我现在对你了解还很少，但我在学习」管理预期
```

### 隐私设计

**全透明原则**：不承诺「绝对安全」，承诺「你永远知道发生了什么」。

- 每次调用云端 AI 时，用户可以看到传出了什么内容、传给谁
- 用户可随时要求「这次对话不要传我的 TELOS」，Agent 降级为普通 AI
- 启动时一次性明确告知数据流向

> 设计者的世界观：普通人的数据早已裸奔在各个平台。这个系统把数据集中在用户自己手里，AI 了解你，且只为你服务——「了解你」本身不是问题，问题是谁在了解你、为谁服务。

### 数据导出

```
RAW_SIGNAL    → JSON / CSV
TELOS         → Markdown（天然可读）
成长事件       → Markdown 时间线
raw_files     → 原始文件打包
```

导出的数据不依赖本系统也能读懂。

### 错误恢复

```
防线 1：版本化
  每次 TELOS 更新前自动保存上一版本，一键回滚

防线 2：置信度标记
  每条 TELOS 内容带 confidence 字段，低置信度内容降权

防线 3：用户主动纠错
  用户说「这条不对」→ 写入 META 校正记录

兜底：一键重置 TELOS 到历史版本
  RAW_SIGNAL 不受影响，Agent 重新提炼
```

### Agent 升级

```
发布新版提炼逻辑
    ↓
所有 RAW_SIGNAL 标记为 processed=false
    ↓
Agent 重新跑完整提炼管道
    ↓
生成新的 TELOS 版本（旧版本保留，用户可对比）
```

## 9. 已识别的设计风险

| 风险 | 缓解方案 |
|---|---|
| 冷启动体验差 | 引导问卷 + 历史数据冷导入 + 预期管理 |
| 提炼逻辑偏离预期 | META 维度存储反馈，三阶段逐步个性化 |
| 本地运行不稳定 | 最终一致性 + 抽象接口支持云端迁移 |
| 用户输入负担 | 多样化输入 + 寄生已有行为 + Agent 主动捕捉 |
| TELOS 维度固定 | 动态维度，Agent 建议用户确认 |
| 隐私与 AI 调用矛盾 | 全透明原则，用户知情选择 |
