# Layers 与 Agent/Scheduler 依赖关系与架构设计

**Date:** 2026-02-04

## Context
在 Huaqi（个人 AI 同伴系统）的架构演进中，目前已经确立了三层架构（接口层、编排层、领域/基础设施层）。为了明确 `agent`（编排层）、`scheduler`（调度层）与 `layers`（领域/基础设施层）之间的交互边界，并确保系统易于进行 TDD（测试驱动开发），需要设计一套合理的依赖关系和通信机制。核心挑战在于：如何在保持严格的层级单向依赖的同时，实现诸如“检测到用户情绪低落需要主动发声”这类由底层领域数据触发上层编排动作的动态交互场景。

## Discussion
在头脑风暴阶段，主要探讨了三种架构依赖方向及其优劣势，并重点评估了它们对 TDD 的友好程度：

1. **严格自上而下的单向依赖（Agent/Scheduler -> Layers）**：
   - 优势：解耦性极强，领域层可完全独立测试（TDD 友好度最高），复用性高。
   - 劣势：底层无法直接触发上层动作，需要引入间接机制。
2. **将编排视为领域能力之一（高内聚黑盒）**：
   - 优势：对外接口简单。
   - 劣势：容易形成“上帝对象”，违反关注点分离，且难以隔离测试，TDD 友好度较低，更适合 BDD。
3. **事件驱动的双向/动态依赖（Agent <-> Scheduler <-> Layers）**：
   - 优势：极其灵活，契合 AI 代理主动发声的特点。
   - 劣势：链路追踪困难，容易导致循环依赖。测试时如果事件隔离不当，容易产生级联副作用。

针对“主动发声”场景，进一步探讨了两种实现路径：
- **Scheduler 轮询（选项 A）**：Scheduler 定时检查 Layers 状态并调用 Agent。优点是保持完美的单向依赖，缺点是不够即时。
- **Layers 抛出事件（选项 B）**：Layers 状态变更时发出事件，Agent 作为订阅者响应。在确认了可以通过 Mock EventBus 实现优雅的 TDD 后，最终选择了此方案。

## Approach
最终确定的方向是：**严格单向依赖结合轻量级事件驱动（EventBus）机制**。

1. **坚持单向依赖**：`Agent` 和 `Scheduler` 可以调用 `Layers`，但 `Layers` 绝不直接依赖 `Agent` 或 `Scheduler`。
2. **事件总线解耦**：当 `Layers`（如领域模型、数据状态）发生关键变更（例如情绪阈值触发）时，只负责向 `EventBus` 发射（emit）事件。
3. **订阅响应**：`Agent`（通过 Hooks 系统）或 `Scheduler` 订阅相应的事件，在接收到事件后执行具体的业务编排逻辑（如生成关怀话语并推送到用户的 Inbox）。
4. **TDD 策略（Mock EventBus）**：测试领域层逻辑时，仅需将 `EventBus` Mock 为一个记录器，验证是否 emit 了正确的事件即可，完全不需要启动复杂的 Agent 状态机或调度器。

## Architecture

### 核心依赖原则
- `Agent` -> 依赖 -> `Layers`
- `Scheduler` -> 依赖 -> `Layers`
- `Scheduler` -> 依赖 -> `Agent`（用于定时触发工作流）
- `Layers` -> 依赖 -> `EventBus`（仅用于抛出状态变更事件）

### 组件交互流（以“情绪低落主动发声”为例）
1. **输入触发**：`Agent` 接收用户日记输入，调用领域层 `AnalysisEngine.analyze(text)`。
2. **领域处理与事件发射**：`AnalysisEngine` 完成情绪分析，更新本地数据。当发现情绪指标达到阈值时，执行 `event_bus.emit("care.mood_critical", {"user_id": uid, "score": score})`。此时 Layers 职责结束。
3. **事件响应**：`Agent` 的 `HooksManager` 监听了该事件。接收到事件后，Agent 后台节点生成关怀消息，存入用户的 Inbox。

### 异常处理与系统恢复
- **异常隔离**：`Layers` 发射事件后不关心订阅者的执行结果。若 `Agent` 生成回复时 LLM API 超时报错，不会影响日记保存和画像更新等核心领域逻辑。
- **持久化与重试**：利用现有的 `layers/data/events/store.py`，核心事件将被持久化到 SQLite。系统重启时，可查询未成功消费的关键事件，通过后台任务重试恢复（Resilience），确保重要通知不丢失。
- **消费端隔离测试**：测试 `Agent` 的事件响应时，无需真实触发完整的日记流程，只需手动调用事件回调函数传入模拟 Event 即可。
