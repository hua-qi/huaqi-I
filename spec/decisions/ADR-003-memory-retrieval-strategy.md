# ADR-003: 记忆检索策略选型

**状态**: 已采纳  
**日期**: 2026-03-29

## 背景

huaqi 的核心体验依赖「记住用户」——对话历史、日记、用户画像随时间持续积累，需要在每轮对话时高效检索相关内容注入上下文。

随着使用时间增长，数据量的预估增长曲线如下：

| 时间节点 | 对话文件数 | 日记数 | 文本总量 |
|------|------|------|------|
| 1 个月 | ~60 个 | 30 篇 | ~200KB |
| 6 个月 | ~360 个 | 180 篇 | ~1.2MB |
| 1 年 | ~720 个 | 365 篇 | ~2.5MB |
| 3 年 | ~2160 个 | 1095 篇 | ~7.5MB |

在此背景下，需要决策：**用什么方式从历史数据中找到与当前对话最相关的内容**。

## 备选方案对比

### 方案一：大模型直接检索

将历史数据直接塞入 prompt，让 LLM 自行定位相关内容。

**失效节点**：数据量超过 context window（约 6 个月后）即完全不可用。超出后必须截断，截断策略本身就是一种检索，逻辑绕回原点。

| 指标 | 评估 |
|------|------|
| 1 个月精度 | 最高（全量上下文） |
| 1 年后可用性 | 失效 |
| token 成本 | 随数据量线性暴增 |
| 延迟 | 5-15 秒，破坏流式体验 |

**结论**：不采纳。短期精度高，但不可持续。

---

### 方案二：LLM 关键词提取 → BM25 全文检索

先用小 prompt 让 LLM 从用户输入提取关键词，再用 BM25 在全量文件中字面匹配。

**核心缺陷**：把语义理解转移给了 BM25，但 BM25 是字面匹配工具，无法桥接「表达不同、意思相同」的 gap。例如用户日记写「情绪低落」，LLM 生成关键词「焦虑、心情差」，BM25 一条都搜不到。

随数据量增长，I/O 扫描成本也线性增加（3 年后全量扫描 2000+ 文件需 2-5 秒）。

| 指标 | 评估 |
|------|------|
| 1 个月精度 | 高 |
| 1 年精度 | 中（字面局限开始显现） |
| 3 年延迟 | 2-5 秒 |
| 额外 LLM 成本 | 每轮 +100 tokens |

**适合场景**：结构化数据（有固定字段、标签）的检索，不适合自由文本日记/对话。

**变体价值**：当搜索对象从原始文件变为「结构化周/月摘要」时，关键词和摘要用词天然对齐，方案三的局限性大幅缩小，可在 L4 层（3 个月以上历史）复用。

---

### 方案三：Embedding + 向量检索（BM25 混合）

离线将历史数据向量化存入 Chroma，查询时只召回语义最相关的 top-K 片段。

| 指标 | 评估 |
|------|------|
| 1 个月精度 | 中 |
| 1 年精度 | 高（语义召回不受字面影响） |
| 3 年延迟 | <200ms（HNSW 索引） |
| 额外 LLM 成本 | 0 |
| 3 年磁盘占用 | ~18MB |

**挑战**：3 年后需要关注向量粒度问题（按文件索引粒度过粗），届时按对话回合分块并定期重索引可解决。

项目已集成 `BAAI/bge-small-zh` + Chroma，基础设施完备，仅需激活。

---

### 方案四：层级摘要索引（Hierarchical Summary）

每天/周/月自动生成摘要，检索时只查摘要树，不碰原始文件。

**适合**：宏观查询（「这个月主要在做什么」），细节查询无法支持。可与方案三组合：摘要树负责宏观定位，向量检索负责细节召回。

---

### 方案五：知识图谱（GraphRAG）

从对话/日记中提取实体和关系构建图，通过图遍历回答关系型查询。

**适合**：关系型查询（「Kaleido 项目的所有讨论」）。实现最复杂，情绪/心理类信息难以图化。数据量到 1 年以上再评估引入价值。

---

### 方案六：结构化提取 + SQL 查询

每次对话结束后 LLM 提取结构化字段（情绪、主题、关键事件）写入 SQLite，通过 SQL 查询。

**适合**：情绪统计、习惯打卡、目标进度类聚合查询。字段设计决定了能查什么，扩展性受限。

---

### 方案七：HyDE（假设文档嵌入）

查询时先让 LLM 生成「假设的历史记录」，用这段文本的向量代替原始问句向量做检索，解决「问句短、文档长」的向量不对称问题。

**定位**：方案三的增强手段，不是独立方案。在向量召回率不足时按需启用。

---

### 方案八：粗召回 + Re-ranking 精排

BM25 或向量检索粗召回 Top-20，再用 Cross-Encoder 或 LLM 精排至 Top-3。

**定位**：方案三的增强层。Cross-Encoder 本地运行，无 API 成本，+50~200ms 延迟。

---

## 决策

采用**分层混合检索**架构，将查询按时间范围路由到不同策略：

```
L1  当前会话      LangGraph messages（完整上下文，in-memory）
L2  近 7 天       直接读 Markdown 文件（~5K token，可直接喂 LLM）
L3  7天~3个月     Embedding 向量检索（BM25 混合，召回 top-3）
L4  3个月以上     LLM 关键词 → 搜索周/月摘要索引
```

**当下优先实现 L1 + L2 + L3**，L4 在数据积累到 3 个月后按需引入。

选择方案三（Embedding + BM25）作为 L3 核心的理由：
1. 项目已有完整基础设施（`bge-small-zh` + Chroma + rank-bm25），仅需激活 `use_vector=True`
2. 语义召回不受用词变化影响，适合自由文本日记/对话场景
3. 3 年后延迟仍在 200ms 以内，不破坏流式体验
4. 零额外 LLM 调用成本

**近期补充**：在 `save_conversation` 节点补全 Chroma 写入逻辑，确保向量库持续被维护。

**远期路线**（数据量超过 1 年后评估）：
- 引入方案六（结构化 SQL）支持情绪/习惯统计类查询
- 引入方案四（摘要索引）优化 L4 宏观检索
- 评估方案七（HyDE）和方案八（Re-ranking）提升 L3 精度

## 结果

- 检索延迟在 3 年数据量下仍可控（<200ms）
- token 成本固定，不随历史数据增长
- 近期可直接基于现有基础设施实现，无新依赖引入
- 架构分层清晰，各层可独立升级替换
